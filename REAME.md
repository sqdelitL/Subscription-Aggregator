# Subscription Aggregator

REST-сервис для управления подписками пользователей и агрегации их стоимости.

## Запуск

Для работы требуется Docker Compose и (опционально) Make.

В `docker-compose.yaml` описаны следующие сервисы:

* `db` — PostgreSQL 13 с healthcheck.
* `app` — приложение, пробрасывает порт `8080`.
* `migrate` — одноразовый контейнер для выполнения миграций БД.

Перед запуском необходимо:

1. Создать файл `.env` по образцу из `.env.template`.
2. Убедиться, что `config.yaml` заполнен корректно.

### Команды Makefile



#### Быстрый запуск без миграций

```bash
make run
```

Эквивалентно:

```bash
docker compose up -d db app
```

После запуска сервис будет доступен по адресу:

```text
http://localhost:8080
```

#### Запуск с автоматическим применением миграций

```bash
make run-migrate
```

#### Сборка образов

```bash
make build
```

Собирает все образы проекта.

Выполняет следующие действия:

1. Поднимает базу данных.
2. Выполняет миграции.
3. Запускает приложение.

Эквивалентные команды:

```bash
docker compose up -d db
docker compose run --rm migrate
docker compose up -d app
```

#### Применить все неприменённые миграции

```bash
make migrate-up
```

Эквивалентно:

```bash
docker compose run --rm migrate /app/migrate-tool up
```

#### Откатить последнюю миграцию

```bash
make migrate-down
```

Эквивалентно:

```bash
docker compose run --rm migrate /app/migrate-tool down
```

#### Создать новую миграцию

```bash
make migrate-new name=имя_миграции
```

Эквивалентно:

```bash
docker compose run --rm migrate /app/migrate-tool new имя_миграции
```

Название используется только для удобства.

## Модель дат

Все даты подписок (начала и окончания) хранятся в PostgreSQL как `DATE`.

При сохранении день всегда принудительно устанавливается в `01` (первый день месяца). Это гарантирует корректное вычисление полных месяцев подписки.

Поля:

* `start_date` — дата начала подписки (первый день месяца).
* `end_date` — дата окончания подписки (также первый день месяца).

Если `end_date` не указана (`NULL`), подписка считается бессрочной.

## Подсчёт стоимости подписок за период

### Endpoint

```http
GET /subscriptions/total-cost
```

### Параметры

| Параметр     | Обязательный |
| ------------ | ------------ |
| start_period | Да           |
| end_period   | Нет          |
| user_id      | Нет          |
| service_name | Нет          |

### Логика подсчёта

1. Из БД выбираются подписки, которые были активны хотя бы один день в заданном периоде с учётом переданных фильтров.
2. На стороне Go для каждой подписки вычисляется количество полных месяцев пересечения с периодом.
3. Итоговая стоимость рассчитывается по формуле:

```text
стоимость = price × количество_месяцев
```

### Почему расчёт выполняется на стороне Go, а не в SQL

* Бизнес-логика остаётся читаемой и легко тестируется.
* Нет жёсткой привязки к функциям конкретной СУБД.
* При росте нагрузки реализацию можно перенести в БД без изменения API, например используя `GREATEST`, `LEAST` и агрегацию по месяцам.

## База данных

При неудачной попытке подключения к БД приложение будет повторять попытки согласно политике, заданной в `data_base.retry` файла `config.yaml`.

## Конфигурация

Перед запуском необходимо:

* заполнить файл `.env`;
* настроить `config.yaml`.

Описание переменных окружения приведено в `.env.template`.

### Основные параметры конфигурации

#### data_base

Настройки подключения к базе данных:

* политика повторных подключений (retry);
* параметры пула соединений.

#### log_level

Поддерживаемые уровни логирования:

* `debug`
* `info`
* `warn`
* `error`

## Линтер

Для статического анализа используется `golangci-lint`.

Конфигурация находится в файле:

```text
.golangci.yaml
```

### Проверки

* `errcheck` — необработанные ошибки.
* `govet` — подозрительные конструкции (кроме `fieldalignment` и `shadow`).
* `staticcheck` — серьёзные баги и возможные упрощения.
* `unused` — неиспользуемый код.
* `gosimple` — избыточные конструкции.
* `ineffassign` — бесполезные присваивания.
* `misspell` — опечатки.

### Запуск

```bash
golangci-lint run
```

### Исключения

Игнорируются:

* директория `vendor`;
* файлы `.pb.go` (сгенерированный код);
* часть проверок в тестах.
